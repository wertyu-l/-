# 异构硬件设备管控与拼接大屏演示系统

## 1. 项目背景

本项目要求实习生开发一个简化的硬件设备管控系统 Demo。管控系统负责用户登录、设备接入与管理、拼接大屏配置和开窗操作；被管控的硬件设备由独立 Java 程序模拟。

系统计划模拟两类异构设备：

1. 分布式节点：使用 HTTP REST 接口和 JSON 数据通信。
2. 嵌入式板卡：使用 UDP 传输和 TLV 二进制数据编码。

项目不要求接入真实硬件，也不要求实现真实视频播放。大屏和窗口通过网页上的逻辑画布展示，模拟设备只需要保存并返回自身状态。

## 2. 实习目标

### 2.1 业务目标

完成以下核心业务链路：

```text
用户登录
  → 添加或发现设备
  → 查看设备状态和能力
  → 创建拼接大屏
  → 绑定设备输出通道
  → 创建、移动、缩放或关闭窗口
  → 模拟设备接收命令并返回状态
```

### 2.2 培养目标

通过项目考查和培养实习生以下能力：

- 理解需求并控制开发范围；
- 在编码前完成基本的软件设计；
- 使用 Java、Spring Boot 和前端技术完成完整功能；
- 理解 REST、UDP、TLV 等不同通信方式；
- 编写有意义的单元测试；
- 使用 Git 和 GitHub 管理代码；
- 编写清楚的构建、运行和使用说明；
- 按计划汇报进展，主动暴露问题。

### 2.3 非目标

- 不复刻公司的正式管控产品；
- 不接入真实生产设备；
- 不实现真实音视频传输与渲染；
- 不要求实现完整权限系统、复杂组织架构和生产级高可用；
- 不使用公司的正式源码、真实协议报文、客户数据和内部地址。

## 3. 建议技术方案

| 模块 | 职责 | 建议技术 |
|---|---|---|
| 管控系统后端 | 登录、设备、大屏、窗口和能力管理 | Java 17、Spring Boot |
| 管控系统前端 | 登录页、设备页、大屏配置和开窗画布 | Vue 3、TypeScript |
| REST 设备模拟器 | 模拟分布式节点 | 独立 Spring Boot 程序 |
| TLV 设备模拟器 | 模拟嵌入式板卡 | 独立 Java 程序、UDP、TLV |
| 数据存储 | 保存用户、设备、大屏和窗口配置 | H2 或 SQLite，避免依赖外部数据库 |

第一阶段只实现 REST 设备，但应预留统一设备驱动接口：

```java
public interface DeviceDriver {

    DeviceInfo getInfo(DeviceEndpoint endpoint);

    DeviceStatus getStatus(DeviceEndpoint endpoint);

    CommandResult openWindow(
            DeviceEndpoint endpoint,
            WindowCommand command);

    CommandResult closeWindow(
            DeviceEndpoint endpoint,
            String windowId);
}
```

后续分别提供：

```text
RestDeviceDriver
TlvDeviceDriver
```

拼接大屏业务只依赖 `DeviceDriver`，不直接判断底层使用 REST 还是 TLV。

## 4. 渐进式开发路线

项目按版本逐步推进。每个版本都必须能够运行和演示，不采用“前几周只写模块，最后一周才整体集成”的方式。

### V0：项目骨架

目标：证明项目可以构建和运行。

功能：

- 创建 GitHub 仓库；
- 创建后端、前端和 REST 设备模拟器；
- 实现简单的用户名密码登录；
- REST 模拟设备提供设备信息和状态接口；
- README 写明开发环境和启动方法。

验收：

- 按 README 可以启动项目；
- 登录后可以看到一台 REST 模拟设备的状态。

### V1：最小管控闭环

目标：完成第一个完整业务流程。

功能：

- 手动添加 REST 设备；
- 查看设备基本信息和在线状态；
- 创建一个 `1×1` 逻辑大屏；
- 将设备输出绑定到大屏；
- 创建和关闭窗口；
- 模拟设备保存并返回窗口状态。

验收：

- 用户可以登录、添加设备、创建大屏和开窗；
- 刷新页面后，可以从模拟设备重新读取窗口状态。

### V2：基础演示版本

目标：形成较完整、可展示的 Demo。

功能：

- 启动多个 REST 设备实例；
- 创建 `2×2` 拼接大屏；
- 为不同屏幕单元绑定设备输出；
- 支持窗口创建、移动、缩放和关闭；
- 保存设备、大屏和窗口配置；
- 设备停止后显示离线。

验收：

- 至少两台模拟设备可以加入同一个大屏；
- 页面操作可以真实改变模拟设备保存的状态。

### V3：设备发现与状态管理

目标：增加设备管控系统的典型特征。

可实现：

- UDP 广播发现设备；
- 区分待添加设备和已管理设备；
- 防止重复添加；
- 简单心跳和在线、离线判断；
- 设备恢复后重新显示在线。

### V4：第二类异构设备

目标：验证统一设备抽象。

可实现：

- 嵌入式板卡 Java 模拟器；
- UDP 请求和应答；
- TLV 编码与解码；
- `TlvDeviceDriver`；
- 使用统一的开窗业务接口控制 REST 和 TLV 两类设备。

TLV 第一版只需要支持固定的少量命令：

- 查询设备信息；
- 查询设备状态；
- 创建窗口；
- 关闭窗口。

请求 ID、超时、重试和通用 Tag 映射可以根据进度逐步增加。

### V5：能力集驱动

目标：让设备能力真正参与业务判断。

能力集可以从简单到复杂逐步增加：

```json
{
  "maxWindows": 2,
  "supportMove": true,
  "supportResize": false,
  "supportOverlay": false,
  "maxResolution": "1920x1080"
}
```

第一阶段只实现一至两条限制，例如：

- 最大窗口数量；
- 是否允许移动或缩放。

后续可以增加：

- 窗口叠加限制；
- 分辨率限制；
- 输入输出通道数量；
- 特定模式下的端口互斥；
- 能力集展示、校验和版本比较工具。

## 5. 难度分级

项目不要求所有功能全部完成。导师根据实习生实际表现动态调整目标。

| 等级 | 目标范围 | 说明 |
|---|---|---|
| 保底目标 | V0 + V1 | 必须形成完整、可运行的最小闭环 |
| 正常目标 | 完成 V2，并从 V3/V4 中选择主要功能 | 达到一般实习项目的预期成果 |
| 良好目标 | 完成 REST 与 TLV 两类设备，并加入简单能力限制 | 能体现协议抽象和业务建模能力 |
| 挑战目标 | SSO、复杂能力约束、CI、Docker、跨设备窗口等 | 只选择一至两个方向深入 |

调整原则：

- 能力一般时，可以减少功能数量，但不能只提交无法运行的零散代码；
- 进度正常时，优先完成第二类设备和简单能力集；
- 能力较强时，再增加可靠性、工程化或复杂业务约束；
- 不要求为了完成全部功能而牺牲代码可读性和基本测试。

## 6. 六周计划

| 周次 | 工作重点 | 主要成果 |
|---|---|---|
| 第 1 周 | 熟悉需求、技术预研、设计和项目骨架 | 个人计划、设计文档 V1、页面原型、V0 骨架 |
| 第 2 周 | 登录、REST 模拟设备、设备添加 | V1 最小闭环前半部分 |
| 第 3 周 | 大屏创建、设备绑定、开窗和状态回读 | 完整 V1，开始 V2 |
| 第 4 周 | 多设备、`2×2` 大屏、移动缩放、上下线 | V2 演示版本 |
| 第 5 周 | 根据能力选择设备发现、TLV 设备或能力集 | V3/V4/V5 中的主要扩展项 |
| 第 6 周 | 修复缺陷、补充测试、整理发布和最终演示 | 发布包、测试说明、使用文档和实习总结 |

每周目标应以“周五可以演示什么”为标准，而不是只描述准备编写哪些代码。

## 7. 设计文档要求

编码前仍然要求完成设计，但不要求实习生分别编写大量正式文档。可以在仓库中使用一份主要设计文档：

```text
docs/design.md
```

设计文档建议控制在能够清楚说明方案的范围内，一般 5 至 10 页即可，至少包含：

1. 项目目标和非目标；
2. 主要业务流程；
3. 系统模块及模块职责；
4. 核心数据模型；
5. 主要 REST 接口；
6. REST 设备和 TLV 设备的控制流程；
7. TLV 报文结构和示例；
8. 计划编写的关键测试；
9. 构建、运行和发布方式；
10. 当前已知风险。

设计要求：

- 第 1 周完成初版并接受导师评审；
- 设计通过后再集中开发核心业务；
- 实现与设计发生明显变化时，及时更新文档；
- 不要求每个小决定都单独编写 ADR，可以在设计文档中增加“设计变更记录”。

技术探针和少量验证性代码可以在设计阶段编写，例如验证 UDP 收发、TLV 编解码或前端拖拽组件，但不能用探针代码代替整体设计。

## 8. GitHub 使用规范

### 8.1 仓库要求

使用实习生自己的 GitHub 公有仓库，仓库建议命名为：

```text
device-control-demo
```

由于仓库公开，必须使用通用化、模拟化的数据和协议：

- 不提交公司正式产品源码；
- 不提交真实设备协议全文和真实报文；
- 不提交客户信息、内部 IP、账号密码、Token 和证书；
- 不提交公司内部制品库、服务器和项目地址；
- 模拟设备名称、能力集和 TLV Tag 应重新设计为 Demo 数据；
- 是否添加开源许可证由导师根据公司知识产权要求确认，实习生不得自行决定。

### 8.2 简化开发流程

不强制实习生使用完整的企业级 Issue、分支保护和 PR 审批流程。

最低要求：

- 使用 Git 正常提交代码；
- `main` 分支中的代码应尽量保持可运行；
- 提交信息能够说明本次修改内容；
- 每完成一个可演示阶段打一个 Git Tag；
- README 随项目进展同步更新。

建议但不强制：

- 较大功能使用 `feature/*` 分支；
- 使用 GitHub Issue 记录重要任务和缺陷；
- 对关键功能提交 Pull Request，由导师做一次代码评审；
- 使用 Milestone 或 Projects 管理每周任务。

推荐的提交格式：

```text
feat: add REST device simulator
fix: reject window outside screen bounds
test: add TLV codec tests
docs: update startup guide
```

避免出现以下情况：

- 实习结束前一次性提交全部代码；
- 提交无法构建的大量半成品；
- 将账号、密码或本机配置提交到仓库；
- 提交大量与本次功能无关的格式化改动。

## 9. 测试要求

项目要求编写单元测试，但不追求生产项目级别的完整覆盖，也不设置强制的整体覆盖率数字。

### 9.1 必须测试的内容

优先测试包含明确规则、容易出错且适合自动化验证的代码：

- 大屏与窗口坐标合法性；
- 最大窗口数量等能力限制；
- 设备驱动选择；
- REST 设备返回异常时的基本处理；
- TLV 编码和解码；
- 错误长度、未知 Tag 等 TLV 异常报文；
- 核心 Service 的主要正常流程和至少一个异常流程。

### 9.2 不作硬性要求的内容

- 不要求每个 Getter、Setter 和简单数据对象都写测试；
- 不要求达到 70%、80% 等整体覆盖率；
- 不要求完整的端到端自动化测试；
- 不要求使用 Testcontainers、WireMock 等额外测试框架；
- 不要求为了覆盖率编写没有有效断言的测试。

### 9.3 测试验收

实习生需要能够说明：

1. 为什么选择这些代码编写测试；
2. 每个关键测试在防止什么问题；
3. 正常、边界和异常情况分别如何验证；
4. 如何在本地通过一条命令运行全部测试。

如果项目生成覆盖率报告，可以作为加分项，但覆盖率数字不作为唯一评价依据。

## 10. 构建与发布要求

“减少环境依赖”的主要目标是：项目不依赖实习生个人电脑、IDE、固定路径、外部数据库和公司内部服务。

### 10.1 必须做到

- 提供清晰、准确的 README；
- 使用 Maven Wrapper 或 Gradle Wrapper，避免依赖本机预装特定构建工具；
- 明确要求的 JDK 和 Node.js 版本；
- 使用 H2 或 SQLite，避免要求验收人员安装 MySQL、PostgreSQL、Redis 等服务；
- 所有端口和地址可以通过配置修改；
- 不硬编码 Windows 或 Linux 的个人目录；
- 提供示例配置，不包含真实密钥；
- 后端可以构建为可执行 Jar；
- 前端可以生成静态发布文件；
- 提供 `start.sh` 和 `start.bat`，或提供同等清楚的启动命令。

### 10.2 最终发布包

建议发布一个压缩包或 GitHub Release，包含：

```text
release/
├── control-server.jar
├── rest-device-simulator.jar
├── tlv-board-simulator.jar       # 如果完成
├── web/                          # 前端静态文件
├── config/
├── start.sh
├── start.bat
└── README.md
```

运行环境只要求安装明确版本的 JRE；不得要求验收人员安装 IDE、Maven、数据库或修改源码。

### 10.3 Docker 和 CI 的定位

以下内容全部调整为选做或加分项：

- Dockerfile；
- Docker Compose 一键启动；
- GitHub Actions 自动构建；
- 自动生成覆盖率报告；
- PR 质量门禁；
- 自动发布 GitHub Release。

如果实习生主线功能完成较早，可以选择其中一项进行实现，不要求全部完成。

## 12. 阶段完成标准

一个阶段达到以下条件即可验收：

- 功能能够实际运行和演示；
- 主要代码已经提交到 GitHub；
- 设计或 README 与实现保持基本一致；
- 关键业务逻辑有必要的单元测试；
- 没有提交账号密码和个人环境配置；
- 已知问题已经记录，不要求所有非关键问题全部解决。

不要求每个阶段都必须创建 Issue、PR、Docker 镜像或 CI 流水线。

## 14. 考核建议

考核重点放在实习生是否能够独立、持续地推进一个完整项目，而不是完成了多少高级工具。

| 考核项 | 建议权重 | 主要观察内容 |
|---|---:|---|
| 业务功能与迭代 | 35% | 是否优先完成闭环、功能是否真实可用 |
| 设计与代码质量 | 25% | 模块划分、接口设计、代码可读性和异常处理 |
| 测试意识 | 15% | 是否为关键规则和协议编写有效测试 |
| 构建与交付 | 10% | README、配置和发布包是否可复现 |
| Git 使用 | 5% | 提交是否持续、清晰，仓库是否整洁安全 |
| 沟通与问题解决 | 10% | 日报周报、问题暴露、分析和复盘 |

CI、Docker、完整 Issue/PR 流程不作为基础考核项，只在实习生完成主线后作为加分依据。
